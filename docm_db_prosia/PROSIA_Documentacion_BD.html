1   Introducción al sistema PROSIA

PROSIA es un Sistema de Ejecución de Manufactura (MES) diseñado para pequeñas y medianas empresas del sector manufacturero. Su objetivo es digitalizar y controlar los procesos productivos: desde la planificación de órdenes hasta el control de calidad y el seguimiento de inventario.


¿Qué problemas resuelve?

Problema del negocioMódulo que lo resuelve
¿Cuánto tiempo tardó cada operario en cada operación?Registro de tiempo (Producción)
¿Cuándo paró la máquina y por qué?Paradas de máquina (Producción)
¿Qué materiales se consumieron en esta orden?BOM + Lotes (Inventario)
¿Este producto pasó el control de calidad?Inspecciones + No conformidades (Calidad)
¿Quién modificó este registro y cuándo?Auditoría de usuario (Analítica)
¿Cuál es la eficiencia del área de ensamble este turno?KPI Snapshots (Analítica)
Alcance actual PROSIA v1 apunta a empresas de una sola planta (PYME). El esquema no incluye multiempresa ni multiplanta. El sistema admite múltiples usuarios, roles y áreas de trabajo dentro de una misma organización.

2   Arquitectura general del schema

2.1 Capas del schema

El schema está organizado en cinco capas que reflejan la jerarquía de dependencias:


CAPA 1 — ENUMs y funciones utilitarias (sin dependencias)
    ↓
CAPA 2 — Tablas catálogo (datos parametrizables, pocos registros)
    ↓
CAPA 3 — Tablas transaccionales de producción (el corazón del sistema)
    ↓
CAPA 4 — Tablas de inventario y calidad (dependen de producción)
    ↓
CAPA 5 — Tablas de analítica y auditoría (observan a todas las demás)

2.2 ENUMs — valores fijos del sistema

Un ENUM es un tipo de dato que solo admite valores predefinidos. Se usa cuando el negocio nunca necesitará agregar nuevas opciones sin cambiar código. PROSIA tiene 10 ENUMs:

2.3 Catálogos vs ENUMs — la decisión clave

Regla de oro del diseño PROSIA Se usa ENUM cuando los valores son universales y no cambian con el negocio (ej: las acciones de auditoría siempre serán Crear/Modificar/Eliminar).

Se usa tabla catálogo cuando el administrador del sistema necesita agregar o cambiar valores sin tocar código (ej: tipos de movimiento de inventario, estados de orden de producción).

ON DELETE SET NULL vs CASCADE vs RESTRICT
CASCADE: si borro el padre, borro los hijos (ej: borrar turno borra sus descansos — lógico).
RESTRICT: no puedo borrar el padre si tiene hijos (ej: no puedo borrar un ítem si tiene órdenes — protege datos).
SET NULL: si borro el padre, el hijo queda huérfano con NULL (ej: borrar usuario no borra la auditoría — conserva evidencia).

9   Sugerencias técnicas Para consultar con instructor

Sobre esta sección Las siguientes sugerencias son mejoras identificadas durante el diseño. No son errores — el schema funciona correctamente sin ellas. Su implementación depende del criterio del instructor y del alcance del proyecto.

S1 — PKs de tablas de alto volumen deben ser SERIAL o GENERATED IDENTITY

ALTA

Tablas afectadas: tab_evento_produccion, tab_auditoria_usuario, tab_sesion_usuario, tab_kpi_snapshot, tab_alerta.


Problema: Estas tablas usan DECIMAL(10,0) con asignación manual de ID (patrón COALESCE + MAX). Ese patrón funciona bien para catálogos donde un administrador inserta 5 registros al año. Pero estas tablas reciben inserciones automáticas constantes: cada acción del sistema genera un evento, cada login genera una sesión, etc. Con múltiples requests simultáneos, dos inserciones podrían calcular el mismo ID y una fallaría.


Solución: Cambiar el PK a SERIAL o INT GENERATED ALWAYS AS IDENTITY. PostgreSQL garantiza IDs únicos y consecutivos sin importar la concurrencia.


Ejemplo del cambio:

-- Antes
id_evento   DECIMAL(10,0) NOT NULL,

-- Después
id_evento   INT GENERATED ALWAYS AS IDENTITY,

S2 — Seguridad: implementar Row-Level Security (RLS) en PostgreSQL

ALTA

Actualmente el control de acceso está modelado en la aplicación (tab_rol, tab_permiso_menu_rol). Esto significa que si alguien accede directamente a la base de datos por fuera de la aplicación, puede leer o modificar cualquier tabla.


Solución: Activar Row-Level Security en PostgreSQL. Permite definir políticas a nivel de base de datos: "el usuario con rol Operario solo puede leer sus propios registros de tiempo". Incluso si se conectan directamente a la BD, las políticas se aplican.


Esta sugerencia se desarrollará en la sesión de seguridad pendiente.

S3 — Agregar índices en columnas de alta consulta (FK y filtros frecuentes)

MEDIA

PostgreSQL crea índices automáticamente solo en PRIMARY KEY y columnas UNIQUE. Las Foreign Keys y columnas usadas en filtros (WHERE) no se indexan automáticamente.


Columnas que se deben indexar manualmente:

  • tab_registro_tiempo.id_usuario — consultas de "tiempo por operario"
  • tab_registro_tiempo.fecha_inicio — reportes por rango de fechas
  • tab_movimiento_inventario.id_item — kardex por ítem
  • tab_auditoria_usuario.id_usuario — historial de acciones de un usuario
  • tab_auditoria_usuario.fecha_accion — reportes de auditoría por período
  • tab_lote.codigo_lote — búsqueda por código (ya es UNIQUE, ya tiene índice)
  • tab_sesion_usuario.token_hash — validación de sesión en cada request (ya es UNIQUE)

Sintaxis: CREATE INDEX idx_reg_tiempo_usuario ON tab_registro_tiempo(id_usuario);

S4 — tipo_registro en tab_registro_tiempo debería ser un ENUM o CHECK más estricto

MEDIA

Actualmente tipo_registro es VARCHAR(20) DEFAULT 'Produccion'. No hay restricción que impida insertar valores inválidos como 'produccion' (minúscula), 'producción' (tilde) o 'Calibracion' (valor no definido).


Opción A — ENUM (recomendada): Crear enum_tipo_registro con valores 'Produccion', 'Preparacion', 'Limpieza' y usarlo como tipo de la columna.


Opción B — CHECK constraint:

CONSTRAINT chk_tipo_registro CHECK (tipo_registro IN ('Produccion', 'Preparacion', 'Limpieza'))

S5 — Considerar particionamiento de tablas de log a futuro

BAJA

Con el tiempo, tab_auditoria_usuario y tab_evento_produccion pueden acumular millones de registros. PostgreSQL soporta particionamiento por rango de fechas: cada mes o año se convierte en una partición separada, lo que mantiene las consultas rápidas sin necesidad de archivar manualmente.


No es urgente implementarlo desde el inicio, pero es bueno diseñar sabiendo que existe esta opción.

S6 — f_anio_actual() debería documentarse como función de conveniencia

BAJA

La función f_anio_actual() encapsula EXTRACT(YEAR FROM CURRENT_DATE)::integer para usarla en el CHECK de tab_maquina. Es válida y funciona correctamente.


La alternativa directa es usar EXTRACT(YEAR FROM CURRENT_DATE) inline en el CHECK, sin crear una función separada. La función agrega una capa de abstracción que puede confundir a quien lea el schema sin leer la sección de funciones primero.


Cualquiera de las dos es válida — es decisión de estilo de código.
